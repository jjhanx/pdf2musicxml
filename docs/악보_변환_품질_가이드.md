# 악보 PDF → MusicXML 품질·호환 복구 가이드

이 파이프라인은 **OCR + Audiveris + 경량 가사 병합**으로 구성되어 있습니다. 그대로 얻은 결과가 인쇄본·상업 악보와 같을 기대는 어렵고, 아래 순서로 단계별 보완하는 것이 현실적입니다.

---

## 서버 배포 후 점검 체크리스트 (단계별)

운영 서버에서 새 커밋을 반영한 뒤, **아래 순서대로** 확인하면 설정 누락과 깨진 동작을 빨리 걸러낼 수 있습니다.

|------|--------|------------------------|
| 1-1 | 저장소 최신 반영: `git fetch` 후 `git pull origin main`(또는 운영 브랜치) | 충돌 없이 최신 커밋. |

| 4-2 | 여전히 깨지면 | 브라우저 종류, nginx가 `Content-Disposition`을 수정하는지, 다운로드 프록시 버퍼 확인. |

### 5단계: 마스킹·텍스트 추출 (2번 이슈 대응 확인)

| 순서 | 할 일 | 기대 결과 / 실패 시 |
|------|--------|------------------------|
| 5-1 | 디버그 ZIP의 **`masked_input.pdf`** | 가사·제목 영역이 흰 블록으로 가려지고 **오선·음표는 최대한 유지**. |
| 5-2 | **`text_data.json`** | `source: "vector"` 항목이 벡터 PDF에서 채워지는지 확인. 이전처럼 OCR 잡음만 과다하면 **PDF가 스캔본**이거나 벡터 레이어가 거의 없을 수 있음. |
| 5-3 | 가사 **찌꺼기가 많이 남음** | `PDF2MXL_VECTOR_OCR_SKIP_THRESHOLD` **하향**(더 많은 페이지에서 OCR). 또는 해당 PDF가 진짜 **이미지만 있는 스캔**이면 벡터 추출 한계 — 소스 PDF 종류 변경 필요. |
| 5-4 | 반대로 **악보선이 깎이거나 과마스킹** | 임계값 **상향** 또는 이슈 재현 페이지만 수동 검토(코드 튜닝은 별도). |

### 6단계: MXL / mxlplayer (3·4·5·6번 이슈 대응 확인)

| 순서 | 할 일 | 기대 결과 / 실패 시 |
|------|--------|------------------------|
| 6-1 | 디버그 ZIP에서 **`masked_input.mvt1.mxl`** 등 **병합 전** 파일만 mxlplayer에 넣어 봄 | **병합 전에도** `realValue` 오류 → **Audiveris MusicXML 4.x vs 뷰어** 가능성↑. MuseScore 등에서 열어 **MusicXML 3.1** 재내보내기 권장. |
| 6-2 | **마디 수**가 실제 악보보다 훨씬 적음 | Audiveris 인식 실패·잘린 파트. **다른 `mvt*.mxl`**를 각각 확인, 마스킹 품질·원본 PDF 해상도 확인. |
| 6-3 | **`*_merged.mxl`** + **`*_merged_lyrics.txt`** | 기본 모드: MXL은 Audiveris 출력과 동일(또는 거의 동일), 가사 문장은 **텍스트 파일**로 확인. `PDF2MXL_INJECT_LYRICS_DIRECTIONS=1`일 때만 구식 방식 주입. |

### 7단계: EasyOCR·버퍼 (기존 이슈)

| 순서 | 할 일 | 기대 결과 / 실패 시 |
|------|--------|------------------------|
| 7-1 | `text_data.json`이 항상 `[]`이고 마스킹 안 됨 | EasyOCR 모델 손상 가능 — `~/.EasyOCR/model` 삭제 후 재다운로드. 서버는 stdout 버퍼를 넉넉히 두었는지(구버전) 로그 확인. |

## 1. 파일명이 깨질 때 (`_�@…`)

- 업로드 시 브라우저/서버가 **파일명 인코딩**을 다르게 해석하면 ZIP·다운로드 이름이 깨집니다. 최신 코드는 UTF-8 우선·NFC·한글·대체 문자를 고려합니다.
- nginx 뒤라면 업로드 본문·응답 헤더가 잘리지 않는지, `Content-Disposition` 전달 방식을 확인하세요.

## 2. 가사·한글: 벡터 PDF는 글리프 추출이 우선

- **텍스트가 포함된 PDF**(Finale/Muse 등에서 내보낸 벡터 PDF)는 **OCR보다 PyMuPDF `get_text` 추출**이 정확합니다.
- 최신 `pdf_text_extractor.py`는 **벡터 텍스트를 먼저 마스킹**하고, 필요한 페이지만 **EasyOCR**로 보완합니다. OCR 후보가 벡터 영역과 **겹침(IoU)**이 크면 버립니다.
- 페이지당 벡터 글자 수가 `PDF2MXL_VECTOR_OCR_SKIP_THRESHOLD`(기본 **40**) 이상이면 해당 페이지 **EasyOCR 생략**.
  - **가사가 덜 지워지면** → 값을 **낮춤**(예: 20~30).
  - **과도한 OCR로 노이즈·선 침범**이 늘면 → 값을 **올림**.

## 3. `realValue` / mxlplayer 오류

- Audiveris **MusicXML 4.0.x** 출력은 일부 뷰어가 완전히 지원하지 않을 수 있습니다. **병합 전** `masked_input.mvt1.mxl`에서도 같은 오류가 나면 **병합기 문제가 아니라 Audiveris·뷰어 호환** 쪽에 가깝습니다.
- **대응**: [MuseScore](https://musescore.org/) 등에서 MXL을 연 뒤 **MusicXML 3.1** 등으로 다시 내보내 mxlplayer에서 시험하세요.

## 4. 마디 수가 줄어 보임 (63 → 15 등)

- **인식 실패**·**파트 매핑**·**뷰어가 특정 파트만 표시**하는 경우가 있습니다. 디버그 ZIP의 **여러 `.mxl` 파트**(mvt1, mvt2…)를 각각 열어 보고, Audiveris 로그·원본 마스킹 PDF를 확인하세요.

## 5. 가사 병합(`*_merged.mxl`)

- 예전 방식은 **첫 마디에 `direction`/`words`를 대량 삽입**해 PDF 좌표를 그대로 넣어, 스펙·뷰어와 어긋지기 쉽습니다.
- **기본값(안전 모드)**: MXL **원본은 그대로 두고**, 추출 문장은 **`<이름>_merged_lyrics.txt`**(같은 폴더, 디버그 ZIP에 포함되도록 서버에서 묶음)로만 둡니다. 가사를 음표에 맞추려면 악보 편집기에서 **수동 `<lyric>` 배치**가 필요합니다.
- 예전 동작이 필요하면 서버에서만  
  `PDF2MXL_INJECT_LYRICS_DIRECTIONS=1`  
  설정 후 Node 프로세스를 **재시작**하세요.

## 6. 권장 워크플로

1. **디버그 ZIP**로 `masked_input.pdf`, `text_data.json`, 각 `.mxl`·`*_lyrics.txt`를 확인합니다.
2. **마스킹**이 악보선을 침범하면 `PDF2MXL_VECTOR_OCR_SKIP_THRESHOLD` 조정 또는 PDF 소스 변경을 검토합니다.
3. **Audiveris만 통과**한 MXL을 MuseScore로 열어 **저장·재내보내기** 후 mxlplayer에 넣습니다.
4. 가사는 **`_lyrics.txt` 참고** 또는 편집기에서 `<lyric>` 요소로 **수동 정렬**합니다.

## 7. 한계

완전 자동으로 **인쇄 품질 가사 정렬**을 맞추는 것은 본 레포 범위를 넘어섭니다. 위 단계로 **재현 가능한 최소 품질**을 맞춘 뒤, 필요 시 Audiveris/MuseScore 커뮤니티 이슈를 참고하는 것이 좋습니다.
