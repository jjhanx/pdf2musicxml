# 악보 PDF → MusicXML 품질·호환 가이드 (Audiveris 단일 파이프라인)

업로드한 PDF를 **그대로 Audiveris**에 넘겨 `.mxl` / `.musicxml`을 만듭니다. **별도 OCR·텍스트 선추출·가사 병합 단계는 없습니다.** 가사·글자는 Audiveris가 인식해 MusicXML에 넣을 수 있는 만큼만 포함되며, 인쇄 품질과 동일을 기대하기 어렵습니다.

---

- Audiveris가 만든 **`.mxl` / `.musicxml`** 파일들

## 마스킹·Audiveris 인식 점검 (웹·API)

Audiveris 오류가 **마스킹된 PDF**에서 이미 악보 기호가 지워졌는지, 아니면 **원본부터** 문제인지 가르는 방법:

1. 변환이 끝난 뒤(또는 `audiveris_review_needed`로 멈춘 상태) 웹 테이블에서 **마스킹·인식 점검**을 엽니다.
2. 페이지를 바꿔 가며 **원본** 열과 **마스킹** 열 PNG를 비교합니다. 버튼을 눌렀을 때 **아무 내용 없이 검게만 보이던** 경우, 예전에는 점검 카드가 `#root`/`.page` 레이아웃이나 레이어(z-index)·overflow에 묻혔을 수 있습니다. 현재는 점검 모달을 **`document.body`로 포탈**해서 띄우며, 우측 **MusicXML(OSMD)** 렌더는 실패해도 PNG 비교 줄만으로 마스킹을 확인할 수 있게 안내 문구로 대체합니다. 미리보기가 **검은 화면**이거나 비면 브라우저에서 PNG URL이 에러인지 확인하고, 같은 작업 폴더의 `.diag-cache/` 래스터 캐시를 지우거나 **렌더 DPI**를 바꿔 재요청해 보세요(서버는 `pdf_diagnostic.py`로 **RGB PNG**를 다시 만들며, 과거 깨진 캐시는 파일명 `-rgb-v2` 변경으로 무효화됩니다). 마스킹에서 기대보다 넓게 하얗게 덮이면 `mask_pdf.py`·검토 UI 분류(제목/가사 등)를 의심합니다.
3. MXL이 이미 이상해도, **Audiveris가 읽기 직전**인지 확인하려면 `GET /api/diagnostic/:jobId/masked-pdf`(또는 웹의 **새 탭에서 PDF 열기**)로 **`masked_input.pdf`** 벡터 원본을 여세요. 업로드 원본은 `GET .../original-pdf`입니다. 쿼리 `download=1`이면 저장용 `Content-Disposition: attachment`입니다.
4. 오른쪽 **MusicXML 미리보기**에서 파트를 골라 Audiveris가 낸 **성부별** 결과를 확인합니다. 악보 페이지와 PDF 페이지는 1:1이 아닐 수 있습니다. 웹 미리보기는 OSMD 크래시를 줄이려 **8바표·선(`octave-shift`) 표기만 제거한 복본**으로 그립니다(다운로드 MXL에는 반영되지 않음). OSMD 렌더는 여전히 **컨테이너 폭이 0에 가깝게 잡히는 순간** 또는 **그 밖의 XML 구조**에서 콘솔에 `SkyBottomLine… width not > 0`, \`realValue\` 등이 보일 수 있습니다. 패널을 넓히거나 다른 파트로 바꿔 보고, 필요하면 **PNG 줄 비교만으로 마스킹 검증**을 진행하면 됩니다.
5. API만 쓸 때: `GET /api/diagnostic/:jobId/summary` → `GET .../page/:n/png?source=original|masked` → 선택 `GET .../masked-pdf` / `.../original-pdf` → `GET .../score-musicxml`.
6. 웹 **마스킹·인식 점검** 안의 **「Audiveris 단계별 실행 (디버깅)」**에서 `-step`(예: `GRID`, `PAGE`)까지 배치 실행해 로그·`.omr`을 받거나, `POST /api/diagnostic/:jobId/audiveris-step-probe`로 동일 동작을 자동화할 수 있습니다(`completed`·`audiveris_review_needed`·`failed` 작업).

**보관:** 완료 후 다운로드를 받아도 같은 `jobId`로 위 API는 **TTL(기본 24시간)** 전까지 동작합니다.

**`mask_pdf.py`:** 검토 타입 **`lyrics`** 는 PyMuPDF **글자 단위 텍스트 리덕**으로, `rawdict`에서 **가사로 보이는 글림만** 집습니다.

**기본**은 각 글자 영역마다 **`fill=False` + 공백(또는 `MASK_PDF_LYRIC_REPLACE_CHAR` 한 글자) 치환** 리덕을 써 MuPDF 기본 **흰색 fill** 같은 사각형이 오선·벡터 음표를 덮지 않게 합니다. 예전처럼 흰 박스로만 덮이게 하려면 **`MASK_PDF_LYRIC_PLAIN_REDACT=1`** (디버깅·호환용).

선택적으로 지우는 글자는 한글·라틴 등(`L*`), 숫자(`Nd`), **문자 형태 숫자·로마(`Nl`)**, 공백(`Zs`), 문장 부호·하이픈(`P*` 등, 가사에서 허용하는 하이픈 유니코드)입니다.

텍스트 리독은 리덕 사각형과 글림 bbox가 겹치면 글림이 지워지는 방식입니다. **`MASK_PDF_LYRIC_MUSIC_SAFE`(기본 켜짐)** 는 **겹치는 면적 ÷ 두 bbox 중 더 작은 면적**이 기본 임계치(`MASK_PDF_LYRIC_MUSIC_MIN_OVERLAP` 기본 `0.13`) 이상일 때만 가사 한 글자 리덕을 생략합니다. 교차만으로 과거처럼 전부 보호하려면 `MASK_PDF_LYRIC_MUSIC_LEGACY_INTERSECT=1` 입니다. `MASK_PDF_LYRIC_CHAR_PAD_PT` 로 부풀리는 영역은 **리덕 annot** 에만 들어가고, 음표와의 겹침 판정은 가능한 한 **좁은 글림 bbox** 를 씁니다. `set_small_glyph_heights`로 bbox를 줄여 겹침을 줄입니다.

가사 블록에서 후보 글림을 만들지 못한 경우 기본은 **아무 것도 안 덮음**이며, 필요할 때만 `MASK_PDF_LYRIC_WHITE_FALLBACK=1` 로 **전 블록 흰 박스** 폴백입니다. **이미지만** 된 가사 줄은 선택 리덕이 불가합니다. `MASK_PDF_LYRIC_SELECTIVE=0`이면 가사도 bbox 전체 흰 박스입니다. **제목·템포 등** 다른 타입은 종전처럼 bbox 단위 처리이며, `MASK_PDF_TEXT_REDACT=1`이면 **비-가사** 벡터 텍스트 구역에 전체 리덕을 시도합니다. PyMuPDF 구버전에서 리덕 kwargs를 못 쓰면 자동으로 흰 박스로 폴백합니다.

## 검토 UI → MusicXML 가사 주입

`scripts/inject_ocr.py`는 Audiveris가 내보낸 **`.mxl`** 안의 해당 **`<part>`**에만, 검토 JSON에 적힌 대로 `<lyric>`을 붙입니다.

동일 파트에 가사 블록이 여러 개면 **페이지·세로·가로 순**으로 이어 붙입니다. 완전 자동 정렬이 어려운 **대표 오선 2줄이 한 파트에 섞인** 악보는, 파트 분리가 명확한 악보에 비해 맞추기 어렵습니다.

### 파트 순번 (`lyricPartIndex`)

Audiveris MXL에는 악기·성부마다 `<part id="P1">`, `<part id="P2">`, … 처럼 **여러 파트**가 **문서에 등장하는 순서**대로 들어 있습니다.

- **1** = 첫 번째 `<part>` (맨 위·첫 스태프에 해당하는 경우가 많음)
- **2** = 두 번째 `<part>`
- **4** = 네 번째 `<part>` — 합창 “4부”가 **네 번째 파트**로 나뉘어 있으면 **4**로 지정합니다. (항상 “3=4부”가 아니므로 악보마다 확인)

즉, “이 가사 블록을 **몇 번째 파트의 선**에 붙일지”를 고르는 값입니다. 헷갈리면 MuseScore 등으로 MXL을 열어 **가사를 넣고 싶은 줄이 몇 번째 파트인지**만 확인하면 됩니다.

### Voice (`lyricVoice`) — 멜로디 줄 (1절/2절과 **다름**)

**한 파트 안**에 MusicXML은 **동시에 울리는 서로 다른 선율**을 `<voice>1</voice>`, `<voice>2</voice>` … 로 구분합니다. (합창에서 같은 시각에 다른 리듬의 두 줄이 있을 때 등)

- **1절·2절 가사**는 이 값이 아니라 검토 UI의 **`lyricVerseIndex`(가사 절)** 로 지정합니다. 병합 시 같은 음표에 `<lyric number="1">`, `<lyric number="2">` … 가 각각 붙습니다.
- **`lyricVoice`**: 위 멜로디 줄 중 **어느 줄**에 음절을 맞출지 (보통 단선 성부는 `1`).
- **`*`**: 해당 파트에서 `<voice>`와 무관하게 문서 순 멜로디 후보에 맞춤.

가사 일람은 **지정한 voice와 같은 번호가 붙은 음표만** 순서대로 사용합니다(절마다 **같은 멜로디**를 처음부터 다시 따라감).

**`inject_ocr.py` 동작 보강:** 다성부 파트에서 Audiveris·일부 편집기는 둘째 성부 음에도 `<chord/>`를 붙이는 경우가 많습니다. 예전에는 chord가 붙은 음을 가사 후보에서 전부 빼 **해당 파트에 멜로디 음이 없는 것처럼** 되어 가사가 통째로 사라질 수 있었습니다. 지금은 **같은 `<voice>`의 화음만** 제외하고, **다른 voice와 겹치는 chord 음**은 가사 후보에 포함합니다. 또 검토에서 지정한 `lyricVoice`에 해당하는 후보 음이 **하나도 없으면** 자동으로 **전체 순서(`*`)** 로 바꿔 stderr에 경고를 남깁니다(Audiveris의 voice 번호와 검토 UI가 어긋난 경우). 인식된 음표가 실제보다 **하나 빠지면** 가사가 한 칸씩 밀릴 수 있으므로, 하이픈(`-`)·**앞쪽 음표 생략**으로 조정하세요.

### 가사 절 (`lyricVerseIndex`)

같은 성부·같은 멜로디에 **1절·2절**처럼 **여러 가사 줄**이 있을 때, OCR 블록마다 **가사 절**을 1, 2, 3… 으로 지정합니다. `inject_ocr.py`는 각 절을 독립적으로 같은 멜로디 음표열에 맞추며, MusicXML에는 `<lyric number="1">`, `<lyric number="2">` … 형태로 공존합니다. **절 번호를 `lyricVoice` 2, 3… 으로 표현하지 마세요** — `lyricVoice`는 멜로디 줄(동시 성부) 전용입니다.

### 앞쪽 음표 생략 (`lyricSkipNotes`)

“**이 가사 블록**을 넣기 전에, 지정한 **voice**에서 **선율 음표**(휴식·화음 덧붙임·그레이스 노트는 제외한 카운트) 몇 개를 **가사 없이 건너뛸지**”입니다.

- **0** = 이 블록의 첫 글자부터, 해당 voice의 첫 선율 음표부터 붙이기 시작합니다.
- **예: 4** = 그 voice의 선율 **네 음**은 가사를 달지 않고 지나간 뒤, **다섯 번째 선율 음**에 첫 글자를 붙입니다.

검토 UI의 **하이픈(`-`)** 은 “글자는 없지만 **음표 하나는 소모**”한다는 뜻입니다. **앞쪽 음표 생략**은 그보다 앞 단계에서 **매핑 카운터 자체를 몇 칸 당기는** 조절에 가깝고, **도입부에 반주만 있거나 성부가 늦게 들어올 때** 첫 음절 위치를 맞출 때 씁니다.

### OCR 블록 신뢰도 (`confidence`)

검토 화면에 표시되는 값은 글자 인식 점수(대략 0–100%)입니다. 낮을수록 역할·텍스트를 다시 확인하는 것이 좋습니다. 신뢰도 임계값 이하일 때만 파이프라인을 멈추고 반드시 확인하게 하는 전용 단계는 아직 없으며, 필요 시 향후 확장할 수 있습니다.

### 한 줄 요약

| 필드 | 한 줄 의미 |
|------|------------|
| 파트 순번 | 어느 **파트**(`<part>`)인지 (문서 순서 1부터) |
| 가사 절 | **1절·2절…** (`<lyric number>`). 멜로디 줄과 별개 |
| 멜로디 줄 (voice) | 그 파트 안 **동시 선율**(`<voice>`). `*` = 문서 순 멜로디 후보 |
| 앞쪽 음표 생략 | 그 줄에서 **몇 개 선율 음표를 건너뛴 뒤** 첫 가사를 붙일지 |

### 템포 (`tempo`)

악보에 인쇄된 BPM 표(예: ♩= 75)를 **템포(BPM)** 로 분류하면 해당 영역이 마스킹되고, 검토에서 수정한 숫자가 우선합니다. `inject_ocr.py`는 **첫 파트 첫 마디**에서 기존 `<sound tempo="…">`·`<per-minute>`를 고치거나, 없으면 `direction`+`metronome`+`sound tempo`를 추가합니다. 재생기(mxlplayer 등)가 **템포 요소를 읽지 않고 120으로만 재생**하는 경우는 플레이어 쪽 설정·지원 범위도 함께 확인하세요.

## OMR 음높이·오선이 크게 어긋날 때 (masked PDF → MXL)

`masked_input.pdf`를 Audiveris에 넣었을 때 **온쉼표가 악보 맨 윗줄로 올라가거나**, **음높이가 실제와 10반음 이상** 어긋나는 현상은 “악보 작법을 모른다”기보다 **오선·간격·클프 인식(SCALE/GRID 이후 단계)**이 한 번이라도 빗나면 그 위에 얹히는 **모든 음표 좌표 해석이 연쇄적으로 깨지는** 경우가 많습니다.

### 왜 이런 식으로 보이나

- **음높이**: OMR은 픽셀上的 위치와 **감지된 오선 간격(interline)**·**클프·조**로 MIDI 높이를 계산합니다. 오선이 마스킹으로 끊기거나 얇게 깨지면, “이 음표가 몇 번째 줄인지”가 틀어져 **옥타브 단위나 클프 단위로** 어긋나기 쉽습니다.
- **온쉼표·다성부**: 긴 온쉼표나 성부가 여럿이면 **어느 스태프·어느 voice에 붙일지**를 엔진이 추정하는데, GRID/파트 분할이 틀리면 **엉뚱한 줄 맨 위**처럼 보일 수 있습니다.
- **막대(stem) 방향**: 일반적으로 음높이가 정해지면 세움막대는 **오선 중간/third**를 기준으로 위·아래로 그리는 관례를 **내보내기 프로그램(MuseScore, OSMD 등)**이 적용합니다. **음높이 자체가 잘못되면** 막대도 “거꾸로”처럼 보일 수 있어, 그만으로 “내부가 악보 상식이 없다”고 단정하긴 어렵습니다.

### 이 저장소 파이프라인에서의 “개입” 범위

- Node 래퍼는 Audiveris를 **`-batch -export`** 로만 호출하며, **내부 피치 판별 루틴에 끼어들 API는 없습니다.**
- 할 수 있는 것은 (1) **입력 PDF 품질·마스킹이 오선을 망치지 않게** 하는 것, (2) Audiveris가 지원하는 **CLI `-constant` / `-option`**으로 일부 기본값을 바꾸는 것, (3) **GUI에서 동일 악보를 열어** GRID 등을 수동 교정한 뒤 내보내기, (4) **결과 MXL을 편집기에서 고친 뒤 교체 업로드**하는 것입니다.

### 권장 대응 순서

1. **마스킹 PDF 점검**: 웹 **마스킹·인식 점검** 또는 `GET .../masked-pdf`로 **오선이 하얀 박스에 덮여 끊기지 않았는지** 확인합니다. 끊겼다면 검토 bbox를 조정하거나, 제목·가사만 덮이도록 영역을 나눕니다.
2. **항상 HITL**: 변환 시 **`pauseAfterAudiveris=true`** 또는 서버 **`AUDIVERIS_PAUSE_ON_WARN=1`** 으로 Audiveris 직후 멈춘 뒤, MXL을 MuseScore 등에서 열어 **틀린 부분만 수정 → `continue-audiveris`로 교체 MXL** 제출하는 흐름이 현실적입니다. **전역 `transposeSemitones`는 “통째로 한 키만 밀린” 경우**에만 맞습니다.
3. **Audiveris GUI 대조**: 같은 파일을 GUI로 열어 **SCALE/GRID 단계** 로그와 경고를 봅니다. 배치와 다른 입력으로 취급되는지(렌더 DPI 등)도 확인합니다.
4. **추가 CLI 인자(고급)**: 서버/CLI에 **`AUDIVERIS_CLI_EXTRA_JSON`** 을 설정해 Audiveris 문서에 나온 **`-constant` Fully qualified name**을 넘길 수 있습니다. JSON **문자열 배열**이며, `--` 앞에 그대로 이어 붙습니다.  
   - 예(개념): `["-constant","org.audiveris.omr.sheet.Scale.defaultBeamSpecification=10"]` — 악보마다 효과가 다르며, 잘못 쓰면 더 나빠질 수 있어 [Constants](https://audiveris.github.io/audiveris/_pages/guides/advanced/constants/)·[Sheet scale](https://audiveris.github.io/audiveris/_pages/guides/main/sheet_scale/)를 먼저 읽는 것이 좋습니다.  
   - `GET /api/health`의 **`audiverisCliExtraArgCount`** 로 주입된 토큰 개수만 확인할 수 있습니다(값 내용은 노출하지 않음).
5. **원본을 바꾸는 것**: 인쇄 PDF가 **벡터가 너무 얇거나** 가사가 오선과 겹치면 OMR 난이도가 급상승합니다. 가능하면 **악보 프로그램에서 다시 PDF 내보내기**(해상도·선 두께)하거나, **스캔본이면 300 DPI 이상** 깨끗한 이미지로 재시도하는 편이 낫습니다.
6. **한계**: 자동으로 “음표마다 피치를 고쳐 넣는” 계층은 없습니다. 상용 OMR(예: PhotoScore 등)이나 **처음부터 편집기에 입력**이 더 빠른 악보도 있습니다.

## Audiveris 직후 수동 보정 (음높이·악보)

**글자·가사 역할을 정하는 Pre-Audiveris 검토**와는 단계가 다릅니다. Audiveris가 낸 **악보(MXL)** 를 들어보고·열어본 뒤, 음높이나 잘못 인식된 음을 조정할 때 이 절을 먼저 참고하세요. 웹 UI의 **Audiveris 결과 보정** 모달에서 안내하는 내용과 같습니다.

`POST /api/convert` 폼에 **`pauseAfterAudiveris=true`** 를 주면, Audiveris가 MXL을 낸 직후 서버가 잠시 멈추고 상태가 **`audiveris_review_needed`** 가 됩니다.

1. **`GET /api/raw-mxl/:jobId`**: 주입 전 원본 MXL을 내려받습니다.
2. MuseScore 등에서 확인·수정한 뒤 **`POST /api/continue-audiveris/:jobId`** 로 진행을 재개합니다.

### 조옮김 (`transposeSemitones`)

`continue-audiveris` 요청(필드 `transposeSemitones`, −24〜24)으로 넘긴 값은 `ocr_meta.json`에 기록되고, **`inject_ocr.py`가 가사·메타를 붙이기 직전에** MusicXML **전곡**의 음높이를 같은 반음 수만큼 크로매틱 이동합니다. 리듬·마디 구조는 유지됩니다. **가사 인식(Pre-Audiveris) 검토 화면에서는 조옮김을 다루지 않습니다.** 서버 API의 `POST /api/review` 본문에 `transposeSemitones`를 넣는 경로는 스크립트·고급 연동용으로만 두어 두었습니다.

- **적합:** 악보가 **통째로** 같은 간격만큼만 밀린 경우(예: 전체가 한 옥타브 위·아래).
- **부적합(흔함):** **일부 마디·일부 성부만** 틀린 경우. 전역 조옮김으로는 고치기 어렵고, 편집기에서 **음표 단위**로 고친 뒤 **교체 MXL**을 올리는 편이 맞습니다.

요청 형식:

- **`application/json`**: `{ "transposeSemitones": 0 }` — 지정한 조옮김만 적용하고 Audiveris가 낸 원본 MXL로 이어가기.
- **`multipart/form-data`**: 필드 `transposeSemitones`, 선택 **파일 필드명 `mxl`** — 교체한 MXL과 조옮김을 함께 전달.

이후 파이프라인(OCR·가사 주입 등)이 이어집니다.

### 음표 단위 보정·부분 오류

UI에 “이 음표만 몇 반음” 같은 **부분 조옮김**은 없습니다. **일부만 틀린 음높이**는 MuseScore 등에서 직접 고친 뒤, 위 **`mxl` 교체**로 넘깁니다.

### Audiveris 음높이가 악기·파트마다 다르게 틀릴 때

합창+피아노처럼 **악기가 여럿**이면, OMR는 **조표·클프·오선 간격**을 한 스태프에서 다른 스태프로 잘못 가져오는 경우가 흔합니다. **단일 멜로디 스태프**(위성부 위주)보다 **그랜드 스태프(피아노)** 쪽이 상대적으로 덜 헷갈리는 출력이 나오기도 합니다. 이는 이 저장소의 Node·Python 레이어가 **Audiveris 내부 알고리즘을 바꿀 수 없어** 완전 자동으로 “보컬만 고치기”는 어렵습니다.

**권장:** 동일 PDF를 **Audiveris GUI**로 열어 문제 장의 `GRIDS`·`SPECIAL` 등 단계 로그와 누락·오인식 음표를 확인합니다. 서버에 **`AUDIVERIS_PAUSE_ON_WARN=1`** 을 켜 두면(또는 UI에서 항상 「Audiveris 직후 멈춤」) 표준 로그에 `WARN`이 한 번이라도 나온 배치 변환에서 **결과 보정(HITL)** 단계로 자동 전환되어, MXL을 연 뒤 위 조옮김·교체 흐름으로 정리할 수 있습니다. 필요하면 `AUDIVERIS_WARN_PATTERN`으로 감지 정규식을 좁히거나 넓힙니다.

## 한글 제목이 "Untitled Score"이고 가사가 비어 있을 때

1. **Tesseract 한국어 데이터**: Audiveris는 글자 인식에 Tesseract를 쓰며, 앱 상수 기본값이 **`eng`**라 한글 **TEXTS** 단계가 제대로 안 타거나 결과가 비는 경우가 많습니다. 이 저장소 API는 기본으로 **`kor+eng`** 상수를 CLI에 넣지만, **`kor.traineddata`가 로컬에 없으면** 인식이 실패합니다. GUI에서 **Tools → Languages**로 Korean을 받아 두세요.
2. **`GET /api/health`**: `audiverisOcrLangEffective`가 `"kor+eng"`이고 `audiverisOcrLangConstantInjected`가 `true`인지 확인합니다. 둘 다 기대와 다르면 환경 변수(`AUDIVERIS_OCR_LANG`)와 재시작을 점검합니다.
3. 여전히 비면 Audiveris 로그에서 **TEXTS** 단계 경고·OCR 오류를 확인하고, PDF에서 가사가 **벡터 글자**인지 **이미지**인지에 따라 난이도가 달라집니다.

## Audiveris가 MXL을 만들지 못할 때 (HTTP 422, WARN·ERS)

출력 폴더에 `.mxl`/`.musicxml`이 **하나도 없으면** 작업이 실패합니다.

- 로그에 **`Can't connect to X11`** / **`java.awt.AWTError`** 가 보이면 Audiveris(Java)가 **X11 디스플레이**에 연결하려다 실패한 것입니다(SSH·PM2 헤드리스 서버에서 `DISPLAY=:0`만 export 된 경우 등). Node가 Audiveris를 띄울 때 **`JAVA_TOOL_OPTIONS`에 `-Djava.awt.headless=true`** 를 합쳐 넣도록 되어 있으니 최신 코드로 배포했는지 확인하고, 여전히 같으면 **`unset DISPLAY`** 후 프로세스를 재시작하거나 `xvfb-run` 등으로 감싸는 방법을 검토합니다.
- 로그에 `WARN [파일이름#10]` 형태가 보이면 `#10`은 보통 **10번째 악보 장(sheet)** 문제를 가리킵니다. 한 장이라도 내보내기가 막히면 결과가 비어 보일 수 있습니다.
- **ERS** 등은 Audiveris 내부 단계 관련 메시지가 로그에 붙는 경우가 많습니다.

**직접 확인할 것**

1. 동일 PDF를 **Audiveris GUI**로 열어 문제 장을 수동으로 점검합니다.
2. `pm2 logs` 등으로 **전체 stdout/stderr**에서 `WARN`, `ERROR`, `ERS`와 장 번호를 확인합니다.
3. PDF가 **저해상도 스캔**이거나 가사·글자가 오선과 겹치면 인식이 불안정할 수 있습니다 — 원본 품질·출처를 검토합니다.

## `realValue` / mxlplayer 오류

Audiveris **MusicXML 4.x** 출력은 일부 웹 뷰어가 완전히 지원하지 않을 수 있습니다.

- **대응**: [MuseScore](https://musescore.org/) 등에서 열고 **MusicXML 3.1** 등으로 다시 내보낸 뒤 mxlplayer에서 시험합니다.

## 마디 수·파트가 줄어 보일 때

인식 실패, 파트 매핑, 일부 `.mxl`만 성공하는 경우가 있습니다. 디버그 ZIP의 **여러 부품 파일**(예: `mvt1`, `mvt2`)을 각각 열어 보고, Audiveris 로그와 원본 PDF를 대조합니다.
